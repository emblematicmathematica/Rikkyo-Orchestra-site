# 立教大学交響楽団ウェブサイト 不具合対応手順書

この手順書は、ウェブサイトに不具合が起きたとき、専門知識がない担当者でも最初の確認と引き継ぎができるようにするためのものです。

## 大切な原則

1. 不具合が起きた直後は、CMSやGitHubを続けて編集しないでください。
2. パスワード、二段階認証コード、復旧コード、サーバー接続情報は、このファイルやGitHubへ書かないでください。
3. 履歴を強制的に消す操作は行わず、必ず変更を打ち消す方法で元に戻してください。
4. 分からない状態で複数人が同時に操作せず、対応担当者を一人決めてください。
5. GitHubのOrganizationは団長と広報担当など二人以上で管理し、各自のアカウントで二段階認証を有効にしてください。

## 症状別の確認方法

### サイト全体が開かない

1. 別の端末や通信回線でも開かないか確認します。
2. GitHubの障害情報を確認します。
3. GitHubの「Actions」を開き、最新の公開処理が赤色になっていないか確認します。
4. 公開処理が進行中の場合は、10分ほど待ってから再確認します。
5. 直前の変更後から開かなくなった場合は、「以前の状態へ戻す」の手順へ進みます。

### レイアウトや色がおかしい

1. ページを再読み込みします。
2. プライベートブラウズで確認します。
3. PCだけ、スマートフォンだけ、または全端末で発生するか記録します。
4. 直前にCSSやHTMLを変更している場合は、その変更を戻す候補にします。

### CMSの内容が反映されない

1. GitHubを開き、CMSからの最新変更が保存されているか確認します。
2. 「Actions」で「Build CMS content」を確認します。
3. 黄色または回転中の場合は処理中なので待ちます。
4. 赤色の場合は、失敗画面のスクリーンショットとURLをWeb担当者へ渡します。
5. 緑色なのに反映されない場合は、公開処理の完了を待ってから再読み込みします。

### 画像が表示されない

1. CMSで画像が選択されているか確認します。
2. GitHubの`assets`フォルダに画像が保存されているか確認します。
3. 画像を削除・改名した直後ではないか確認します。
4. CMSの自動更新が成功しているか確認します。

### YouTubeが表示されない

1. YouTube上で元の動画が公開中か確認します。
2. PC内のHTMLファイルではなく、公開サイトで確認します。
3. YouTubeまたはGitHubに障害がないか確認します。
4. 動画そのものが非公開・削除済みの場合は、CMSではなくYouTube側の対応が必要です。

### お問い合わせメールが送れない

現在のフォームはメール作成画面を開く方式です。利用者の端末にメールソフトが設定されていない場合は送信できません。レンタルサーバー契約後、PHPフォームを実装した時点で、以下を追記します。

- [ ] PHPフォームの停止方法
- [ ] SMTP接続の確認方法
- [ ] 迷惑送信対策の確認方法
- [ ] メール送信ログの場所
- [ ] 送信失敗時の代替連絡先

## 以前の状態へ戻す

GitHubには変更履歴が残っています。不具合の原因が直前の更新だと分かる場合は、その変更を打ち消す新しい履歴を作ります。

1. 不具合が始まった日時を確認します。
2. GitHubの変更履歴から、不具合直前の変更を特定します。
3. Web担当者へ、対象の変更番号と不具合の記録を渡します。
4. Web担当者は`git revert`を使って対象変更を打ち消します。
5. 強制push、履歴削除、`reset`は行いません。
6. 公開処理の完了後、PCとスマートフォンで復旧を確認します。

## 緊急用メンテナンスページ

リポジトリ直下の`maintenance.html`に、サーバー障害や長時間の修正時に表示するためのページを用意しています。

現在のGitHub Pagesでは、担当者が公開設定を変更せずに自動で切り替える仕組みはありません。必要になった場合は、Web担当者が状況を確認してから切り替えます。レンタルサーバー契約後に、管理画面やサーバー設定から安全に切り替える方法をこの欄へ追記します。

- [ ] 通常サイトへ戻す方法
- [ ] 切り替え担当者

## セキュリティ上の問題が疑われる場合

次の場合は、通常の表示崩れではなくセキュリティ対応として扱います。

- 身に覚えのないページや広告が表示された
- GitHubやCMSに知らない変更がある
- パスワードや認証コードを他人に見られた
- 公開してはいけない個人情報が掲載された

対応順序：

1. 該当画面と変更履歴を記録します。
2. GitHub、CMS、メール、レンタルサーバーの追加操作を止めます。
3. 団長とWeb担当者へ連絡します。
4. 漏えいが疑われるアカウントのパスワードを変更し、他のログイン状態を解除します。
5. 公開してはいけない情報を非公開にします。
6. 必要に応じて、影響を受けた方への連絡を検討します。

- 団長：団内限定資料に記載
- Web主担当：団内限定資料に記載
- Web副担当：団内限定資料に記載
- レンタルサーバー契約担当：契約後に団内限定資料へ記載
- ドメイン更新担当：取得後に団内限定資料へ記載
